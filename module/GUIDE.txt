VHS REWIND EFFECT
Game Integration Guide
Amiga 68000 Assembly  •  OCS/ECS Chipset
Overview
This guide explains how to integrate the VHS cassette rewind screen effect into an existing Amiga game. The effect simulates old VHS tape artefacts: palette desaturation, horizontal scan-line tearing, noise static, vertical roll, and chroma smear. It is written in 68000 assembly and targets the OCS/ECS chipset.

The integration approach treats the effect as a self-contained module with a clean public API. Your game calls a small set of routines to initialise, trigger, advance, and shut down the effect. The effect takes temporary control of the copper list, blitter, and palette while it runs, then hands everything back cleanly on completion.

STEP 1Understand What the Effect Needs
The effect requires exclusive control of three hardware resources while it runs:
• The copper list — installs per-scanline bitplane pointer patching for horizontal tearing
• The blitter — used for noise overlay, vertical roll, and chroma smear operations
• The palette (COLOR00–COLOR31) — applies the washed-out VHS colour transform

Your game already owns all three. The integration strategy is to pause your game's use of these resources for the duration of the effect, hand control to the effect module, then hand it back cleanly when the effect finishes.
? The effect never modifies your game's original screen content. It works on its own internal work buffers and restores your copper list pointer automatically when done.

STEP 2Restructure the Effect as a Linkable Module
The effect file needs to be split into a module with a clean public interface. Make the following changes to vhs_effect.asm:

Remove from vhs_effect.asm
• Start: entry point and all OS startup code
• AllocChipMem / FreeChipMem — your game manages memory
• TakeOverHardware / RestoreHardware — your game owns the hardware
• MainLoop — your game has its own loop
• The VBlank interrupt handler installation — your game has its own

Add XDEF declarations for the public API
    XDEF    VHS_Init        ; one-time setup    XDEF    VHS_Free        ; release chip RAM    XDEF    VHS_StartEffect ; begin the effect    XDEF    VHS_StopEffect  ; end the effect    XDEF    VHS_DoFrame     ; call once per VBlank    XDEF    VHS_IsActive    ; query if effect running    XDEF    VHS_StateActive ; exported byte flag
All internal subroutines (FillNoisePlane, BuildShiftTable, DoVerticalRoll, DoNoiseOverlay, DoChromaSmear, BuildEffectCopperList, WaitBlit, etc.) require no XDEF — they remain private to the module.

STEP 3Define the Public API
Each public routine has a defined register contract. Follow these conventions strictly so the effect integrates safely with your existing game code.

VHS_Init — call once at game startup
; Input:  a0 = pointer to your screen bitplane 0 (chip RAM);         a1 = pointer to your 32-colour palette (dc.w x 32);         d0 = screen width in pixels  (320);         d1 = screen height in lines  (your visible height); Output: d0 = 0 on failure (not enough chip RAM);             = 1 on success; Saves:  d2-d7, a2-a6 (Exec ABI convention)
VHS_Free — call at game shutdown
; Input:  nothing; Output: nothing; Action: frees all chip RAM allocated by VHS_Init
VHS_StartEffect — begin the effect
; Input:  nothing; Output: nothing; Action: snapshots current hardware palette into NormalPalette,;         computes VHSPalette, sets state to RWIN_IN (fade-in begins)
VHS_StopEffect — end the effect
; Input:  nothing; Output: nothing; Action: sets state to RWIN_OUT (fade-out begins);         VHS_DoFrame will return d0=0 when fully stopped
VHS_DoFrame — call once per VBlank
; Input:  nothing (effect uses its own internal VarBase); Output: d0 = 1 if effect still active;             = 0 if effect fully stopped (your copper restored); Action: runs one complete frame:;           updates state machine;           fills noise plane (LFSR);           builds line shift table;           runs blitter: vertical roll, noise overlay, chroma smear;           builds per-scanline copper list;           applies palette to hardware;           installs copper list into COP1LC + strobes COPJMP1; Note:   when d0=0, your copper list is automatically restored
VHS_IsActive — query effect state
; Output: d0 = 1 if effect is running or fading;             = 0 if fully stopped; Also:   VHS_StateActive (byte) — test directly to avoid call overhead


STEP 4Chip RAM Requirements
The effect allocates its own work buffers from chip RAM during VHS_Init. Your game must have this much chip RAM free at the point VHS_Init is called:

BufferSizePurposeEffect work buffer A51,200 bytesBlitter draw target — effect renders into this each frameEffect work buffer B51,200 bytesDisplay buffer — copper points here while effect runsClean source buffer51,200 bytesRead-only copy of your screen — never written by the effectNoise plane10,240 bytesLFSR pseudo-random noise source for staticCopper list A12,288 bytesEffect copper list (ping buffer)Copper list B12,288 bytesEffect copper list (pong buffer)TOTAL~189 KBMust be in chip RAM
Check available chip RAM before calling VHS_Init:
    move.l  #189000,d0    move.l  #MEMF_CHIP,d1    jsr     LVO_AvailMem(a6)    ; d0 = available chip RAM    cmp.l   #189000,d0    blt     .not_enough_ram
? If chip RAM is very tight, you can eliminate one work buffer by disabling double-buffering inside the effect. This causes a one-frame flicker on the first effect frame but saves 51KB.

STEP 5Your Game's Screen Pointer Convention
Since you don't double-buffer, you have a single screen. The effect needs to know where your bitplanes are. Pass bitplane 0's base address to VHS_Init. The effect computes planes 1–4 as:

Plane 0: your_screen + 0Plane 1: your_screen + PLANE_SIZE    (= 10,240 bytes)Plane 2: your_screen + PLANE_SIZE*2Plane 3: your_screen + PLANE_SIZE*3Plane 4: your_screen + PLANE_SIZE*4
Your planes must be contiguous, non-interleaved, in chip RAM. Example call:
    lea     MyScreen,a0         ; bitplane 0 base    lea     MyPalette,a1        ; 32-word palette table    move.w  #320,d0             ; screen width    move.w  #256,d1             ; screen height    jsr     VHS_Init    tst.l   d0    beq     .vhs_init_failed
Interleaved Bitplanes
If your game uses interleaved bitplanes (all planes for one scanline stored together), use Option A below — it is strongly recommended over modifying the copper builder:

• Option A (recommended): At VHS_Init, use the blitter to de-interleave your screen into the effect's clean source buffer. One-time cost, effect runs unchanged.
• Option B (advanced): Modify BuildEffectCopperList to use BPL1MOD/BPL2MOD to handle interleave stride instead of per-line pointer patching. More complex, more memory efficient.

STEP 6Wiring Into Your VBlank Handler
This is the core integration point. Add a VHS hook to your existing VBlank interrupt handler. The check is a single byte test, so the overhead when the effect is not running is negligible:

YourVBlankHandler:    movem.l d0-d7/a0-a6,-(sp)     ; ... your existing interrupt acknowledge ...    ; ... your existing game logic ...    ; ... your normal copper list update (skip if VHS active) ...     ; ?? VHS EFFECT HOOK ?????????????????????????????????????????    tst.b   VHS_StateActive         ; fast byte test — no call needed    beq     .no_vhs_effect     jsr     VHS_DoFrame             ; runs effect, installs copper    tst.l   d0                      ; d0=0: effect finished    beq     .vbl_done               ; copper already restored by DoFrame    bra     .vbl_done               ; d0=1: effect copper is active .no_vhs_effect:    ; ... your normal copper list update here ...    ; ... move.l  MyCopperList,d0 etc ... .vbl_done:    movem.l (sp)+,d0-d7/a0-a6    rte
? VHS_DoFrame calls WaitBlit before every blitter operation. If your game left the blitter busy, DoFrame will wait politely. Call WaitBlit once yourself before using the blitter again in subsequent frames.



STEP 7Saving and Restoring Your Copper List
When the effect starts it installs its own copper list into COP1LC. When it ends it must restore yours. Pass the address of your copper list pointer variable to VHS_Init so the effect can save and restore it automatically:

    lea     MyScreen,a0    lea     MyPalette,a1    lea     MyCopperList,a2     ; address of YOUR copper list pointer    move.w  #320,d0    move.w  #256,d1    jsr     VHS_Init
Inside VHS_DoFrame, when the state machine returns to STATE_NORMAL and the effect is fully done, the last action before returning d0=0 is:
    ; Restore caller's copper list (executed inside VHS_DoFrame)    move.l  (SavedCopListPtr),d0    swap    d0    move.w  d0,CUSTOM+COP1LCH   ; high word first (OCS requirement)    swap    d0    move.w  d0,CUSTOM+COP1LCL    move.w  #$0000,CUSTOM+COPJMP1  ; strobe: copper reloads immediately
Your game's display snaps back cleanly in the same VBlank that the effect finishes. There is no visible frame of corruption during the handoff.

STEP 8Palette Handling
Your game's palette almost certainly changes during gameplay. VHS_Init stores a palette pointer but does not snapshot it — the snapshot happens at VHS_StartEffect time, capturing whatever palette is active at the moment the effect is triggered. This ensures the washed-out VHS colours are derived from the correct in-game palette.

Internally, VHS_StartEffect performs this sequence:
VHS_StartEffect:    ; 1. Snapshot current hardware palette    lea     CUSTOM+COLOR00,a0    lea     NormalPalette,a1    move.w  #31,d7.snap:    move.w  (a0)+,(a1)+    dbra    d7,.snap     ; 2. Derive VHS palette from snapshot    bsr     ComputeVHSPalette   ; desaturate, darken, tint blue-green     ; 3. Begin effect    move.w  #STATE_RWIN_IN,VAR_STATE(a5)    move.w  #0,VAR_FADECOUNT(a5)    rts
The VHS palette transform applies: 25% saturation reduction, 60% brightness, and a slight blue-green tint to simulate VHS chroma bleed. These parameters are defined as equates and can be tuned to taste.
? Do not call VHS_StartEffect while the effect is already running — check VHS_IsActive first.

STEP 9Interleaved Bitplane De-interleave (Option A)
If your game uses interleaved bitplanes, add this blitter routine to VHS_Init. It runs once and converts your interleaved screen into the contiguous non-interleaved layout the effect expects:

; De-interleave: interleaved source ? contiguous dest; Input:  a0 = interleaved source (your game screen);         a1 = contiguous dest    (effect clean source buffer);         d0 = num planes (5);         d1 = screen height in lines (256) DeInterleave:    movem.l d0-d7/a0-a4,-(sp)    move.w  d1,d6               ; line counter    move.w  d0,d5               ; plane counter saved .plane_loop:    move.l  a0,a2               ; source: interleaved base    moveq   #0,d4               ; current plane index     move.w  d6,d3               ; lines to copy    subq.w  #1,d3               ; for dbra .line_loop:    ; copy one row from interleaved plane d4    ; src offset = line*(planes*stride) + plane*stride    ; This is simplest done with a straightforward byte loop    ; or a blitter copy with appropriate source modulo.    ; See Hardware Reference Manual section on blitter modulo.    dbra    d3,.line_loop     addq.w  #1,d4    cmp.w   d5,d4    blt     .plane_loop     movem.l (sp)+,d0-d7/a0-a4    rts
? The exact modulo calculation depends on your interleave stride. For 5-plane 320-wide interleaved: source modulo = (5-1)*40 = 160 bytes. Set BLTAMOD = 160 when blitting each plane row.

STEP 11Blitter Contention
Your game may use the blitter for sprite rendering, map scrolling, or other effects. VHS_DoFrame uses the blitter intensively across multiple operations per frame. Follow these rules to avoid conflict:

• Call VHS_DoFrame after all your game's blitter operations in the VBlank handler — it will then have the blitter to itself for the rest of the VBlank period.
• VHS_DoFrame calls WaitBlit before every blitter setup. If your game left the blitter busy it will wait politely — your code will not corrupt an in-progress blit.
• After VHS_DoFrame returns, call WaitBlit once before using the blitter again in the next frame to ensure the last effect blit has completed.
• Do not enable DMAF_BLITHOG (blitter nasty bit) in your game while the effect is running — it can prevent the CPU from servicing the VBlank interrupt in time.

; Correct VBlank ordering:     ; 1. Your game blitter ops (sprites, scroll, etc.)    bsr     YourBlitterOps     ; 2. VHS effect (calls WaitBlit internally before each op)    tst.b   VHS_StateActive    beq     .no_vhs    jsr     VHS_DoFrame.no_vhs:     ; 3. Before next frame's blitter use, wait for effect's last blit    bsr     WaitBlit
STEP 12Recommended Trigger Points in a Game
The effect works best when triggered at natural narrative breaks. The fade-in and fade-out are each 12 frames (~240ms at 50Hz) by default — adjust FRAMES_FADE_IN and FRAMES_FADE_OUT equates to taste.

Game EventEffect CallLevel completeVHS_StartEffect ? wait for peak ? swap level data ? VHS_StopEffectGame overVHS_StartEffect ? auto-stop after N frames ? show game over screenPause menu openVHS_StartEffect on pause, VHS_StopEffect on unpauseCutscene transitionVHS_StartEffect ? swap scene mid-effect ? VHS_StopEffectSave/load screenVHS_StartEffect before, VHS_StopEffect after load completesBoss introductionBrief VHS_StartEffect for 30 frames then auto-stop for dramatic entry
Level Transition — Ideal Sequence
This pattern hides the level load behind the peak distortion, giving a seamless rewind-to-next-level feel:

1. Game detects level complete condition
2. Call VHS_StartEffect — screen begins to corrupt and distort
3. While effect is running: load next level's tile data, sprites, map into memory
4. Poll VHS_IsActive — wait until effect reaches full distortion (STATE_RWIN_FAST)
5. At peak distortion: swap your game screen content to next level's initial frame
6. Call VHS_StopEffect — screen fades back in revealing the new level

? The ~240ms fade-in at 50Hz gives approximately 11 frames of loading time before the screen reaches peak distortion. This is typically enough for simple level data loads on stock Amiga hardware.
